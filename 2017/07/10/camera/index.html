Camera

本文主要总结 Camera 类的使用方法，通过类图、时序图深入学习 Camera。

Camera 使用
// 1. 创建 Camera 实例
Camera mCamera = Camera.open();

// 2. 配置 Camera 属性参数
Camera.Parameters mParameters = mCamera.getParameters();
Size size = getCurrentSize(mParameters, height, width);
mParameters.setPreviewSize(size.width, size.height);
mParameters.setAntibanding(Camera.Parameters.ANTIBANDING_AUTO);
mParameters.setFocusMode(Parameters.FOCUS_MODE_CONTINUOUS_PICTURE);
List<Area> areas = new ArrayList<Area>(){};
areas.add(new Area(new Rect(100, 100, 200, 200), 1));
mParameters.setFocusAreas(areas);
mParameters.setPictureSize(640, 480);        
mCamera.setParameters(mParameters);

// 3. 设置 Camera 预览方向
mCamera.setDisplayOrientation(90);

// 4. 设置 Camera 预览界面
mCamera.setPreviewTexture(surface);

// 5. 设置 Camera 预览回调缓冲区
mBuffer = new byte[size.width * size.height * 3 / 2];
mCamera.addCallbackBuffer(mBuffer);
mCamera.setPreviewCallbackWithBuffer(this);

// 6. 开启 Camera 预览
mCamera.startPreview();

// 7. 关闭 Camera 预览
mCamera.stopPreview();

// 8. 释放 Camera 资源
mCamera.release();
拍照

在 Camera 开启预览后，可以调用 takePicture() 进行拍照操作，需要实现 shutterCallback 和 PictureCallback 回调接口来处理拍照按钮事件和照片数据流。

// 1. Camera 拍照
mCamera.takePicture(shutterCallback, pictureCallback, jpegCallback);

// 2. 继续预览
mCamera.startPreview();
录像

在 Camera 开启预览后，调用 unlock() 允许媒体进程访问相机，通过 setCamera(Camera) 将相机权限交给 MediaRecorder 来进行录像，最后调用 reconnect 重新获得相机使用权。

// 1. 
mCamera.unlock()；

// 2. 设置 MediaRecorder
MediaRecorder recorder = new MediaRecorder();
recorder.setCamera(mCamera);
...

// 3. 
mCamera.reconnect();
Camera 类图

Camera 中定义了在其生命周期中需要的内部类和接口

内部类：
  • CameraInfo：保存 Camera 的前后置，图片方向以及能否取消拍摄声音
  • Size：图片大小
  • Face：脸部识别信息
  • Area：为曝光、白平衡、自动聚焦设置区域信息
  • EventHandler：处理各个过程产生的回调，调用相应的回调函数
  • Parameters：设置 Camera 属性参数
接口：
  • ErrorCallback：出错回调接口
  • PreviewCallback：预览回调接口
  • ShutterCallback：拍摄回调接口
  • PictureCallback：拍照后图片回调接口
  • AutoFocusCallback：自动聚焦回调接口
  • OnZoomChangeListener：焦距变化监听器

补充

  • IInterface：为 client/server 端提供接口，它的子类有 BpInterface 和 BnInterface。

  • IBinder：定义了 binder IPC 的通信协议。BBinder 和 BpBinder 均为IBinder的子类，是在IBinder协议框架内进行的收和发操作，构建了基本的 binder IPC 机制。

  • BpRefBase：client 端在查询 ServerManager 获得所需的 BpBinder 后，BpRefBase 负责管理当前获得的 BpBinder 实例。

  • BpInterface：如果 client 端想要使用 binder IPC 与 Service 通信，那么首先会从 ServiceManager 处查询并获得 server 端 service 的 BpBinder，在 client 端，这个对象被认为是 server 端的远程代理。为了使 client 能够像本地调用一样调用一个远程 server，server 端需要向 client 提供一个接口，client 在这个接口的基础上创建一个 BpInterface，使用这个对象，client 的应用能够像本地调用一样直接调用 server 端的方法。而不用去关心具体的 binder IPC 实现。

    // BpInterface 的原型：
template<typename INTERFACE>
class BpInterface : public INTERFACE, public BpRefBase

    BpInterface 是一个模板类，当 server 提供了 INTERFACE 接口(形如 IXXXService)后，通常会继承 BpInterface 模板类实现一个 BpXXXService：

    class BpXXXService: public BpInterface<IXXXService>

    实际上 BpXXXService 实现了双继承 IXXXService 和 BpRefBase，这样既实现了 service 中各方法的本地操作，将每个方法的参数以 Parcel 的形式发给 binder driver；同时又将 BpBinder 作为了自己的成员来管理，将 BpBinder 存储在 mRemote 中，通过调用 BpRefBase 的 remote() 来获得 BpBinder 指针。

  • BnInterface：BnInterface 也是一个模板类。在定义 native 端的 service 时，基于 server 提供的 INTERFACE 接口(IXXXService)，通常会继承 BnInterface 模板类实现一个 BnXXXService，而 native 端的 service 继承自 BnXXXService。BnXXXService 定义了一个 onTransact 函数，这个函数负责解包收到的 Parcel 并执行 client 端的请求的方法。

    // BnInterface 的原型是：
template<typename INTERFACE>
class BnInterface : public INTERFACE, public BBinder
// BnXXXService 形如：
class BnXXXService: public BnInterface<IXXXService>

    IXXXService 为 client 端的代理接口 BpXXXService 和 server 端的 BnXXXService 的共同接口类，这个共同接口类的目的就是保证 service 方法在 C/S 两端的一致性。每个 service 都可视为一个 binder，而真正的 service 端的 binder 的操作及状态的维护就是通过继承自 BBinder 来实现的。BBinder 是 service 作为 binder 的本质所在。那么，BBinder 与 BpBinder 的区别是什么呢？BpBinder 是 client 端创建的用于向 server 发送消息的代理，而 BBinder 是 server 端用于接收消息的通道。他们的代码中虽然均有 transact 方法，但两者的作用不同，BpBinder 的 transact 方法时想 IPCThreadState 实例发送消息，通知其有消息要发送给 binder driver；而 BBinder 则是当IPCThreadState 实例收到 binder drive r消息时，通过 BBinder 的 transact 方法将其传递给它的子类 BnXXXService 的 onTransact 函数执行 server 端的操作。

  • Parcel：Parcel 是 binder IPC 中的最基本的通信单元，它存储 C/S 间函数调用的参数。Parcel 只能存储基本的数据类型，如果是复杂的数据类型的话，在存储时，需要将其拆分为基本的数据类型来存储。

C/S 架构

Camera C/S binder IPC 通信采用的是半双工通信的方式。client 端提供的接口为 ICameraClient，分别通过继承 BpInterface 与 BnInterface 模板类的方式实现代理 BpCameraClient（server 端的 client 代理）与 client 端本地 BnCameraClient，Camera 为本地具体的实现；server 端提供的接口为 ICameraServer，分别通过继承 BpInterface 与 BnInterface 模板类的方式实现代理 BpCameraService（client 端的 server 代理）与 serve r端本地 BnCameraService。
